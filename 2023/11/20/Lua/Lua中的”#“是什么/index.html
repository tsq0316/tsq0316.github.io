Lua中的#到底等于多少

发表于 2023-11-20 更新于 2023-11-29 分类于 Lua 阅读次数： 本文字数： 4.4k 阅读时长 ≈ 4 分钟

在使用 Lua 时，我们经常见到这种用法：#t（t是表名）或是 #str（str是字符串名），这种写法可以用于求表的长度，或是求字符串的长度，但我们经常发现得到的值并不是我们想要的结果。网上对于 # 的解释很多，但常常有错误的情况，例如说 table 的第一个关键字不是数字，那么表长就是 0。事实真的如此吗？我们可以来简单探究一下。

注意，不同版本的 lua 也许对 # 的实现有更改。

求字符串长度

对于求字符串长度，有两种方式：使用 string.len 和 # ，在大多数时候这两种方式是等效的，# 是一种更加简洁的语法糖，两者的计算都是从下标 1 开始的，所以得到的结果就是字符串的长度。示例：

需要注意的是，在输出字符串时，会在遇到的第一个 null 字符处停止（ASCII 0），但是使用 # 和 string.len 计算字符串长度时，在 null 之后的字符也会被计入长度。

求表长度

使用 # 求表长度，需要注意些什么？网上大部分的解释都不够准确或者全面，我们可以通过举例来一一求证。

对于字典

由以上例子可以观察到:

  • 对于索引值为 1~n 的字典，只要没有键的中断，# 返回的长度就是 n，无论插入的顺序。
  • 对于索引值小于 1 的键值对，长度没有计入。
  • 对于中断了的正整数索引，会随机返回一个中断的索引作为 # 的结果。简单来说就是，索引 n 是一个非 nil 值，索引 n + 1 为 nil，那么就返回 n。如果有多个这样的 n，那么就随机返回一个（例如table2返回了2，table3返回了4）。

数组和字典的混合

为什么 # 返回的表长度会是 3 呢？在我本来的设想中，”abc” 和 true 分别对应于默认索引 1 和 2，索引 3 和 5 是有值的，那么对于 “world”, 如果给它分配一个默认索引 4，表长就是 5；如果给它分配一个索引 6，表长就是 3。难道说有效计入表长度的值是 “abc”, true, 3.14 吗？我选择用两种方式打印看看。

由上述示例可以看到，对于混合了数组、字典两种形式的 table，在初始化表值时，会先把数组部分分配连续的从 1 开始的索引，如果字典部分有相同的索引，并不会覆盖掉默认索引的值（如果是在初始化结束后，在表外赋值，例如```table5[3] = 3.14，依然会被覆盖掉）。所以对于字典、数组混合的情况，# 的返回值有这样一些考虑：

  • 优先给数组部分的值分配从 1 开始的整数索引；
  • 分配给了数组部分的索引，初始化时不会再分配给字典部分的值；
  • 非整数索引不考虑计入；
  • 最后对于所有的正整数索引，如果有多个中断值，那么可能会返回其中任意一个。例如最后的索引值包含1，2，3，5，7，那么可能返回的就是3，5，7。

带元方法的表

如果表使用了 __len 元方法，# 操作符会调用该元方法来获取长度。用户定义的元方法可能返回任意值，而不一定是表的实际长度。
